RAG Chatbot Platform

A Multi-Backend Retrieval-Augmented Generation System

Built with FastAPI PocketBase Caddy Google Cloud

The Problem

  • Data-grounded AI — Organizations need chatbots that answer from their own data, not generic training data
  • Multiple retrieval strategies — Different data sources (corpora, search engines, vector DBs) require different backends
  • Easy multi-team deployment — Need to spin up instances across VMs without re-engineering each one
  • Admin control without code changes — System prompts, models, and retrieval settings should be editable at runtime
  • Two audiences — End-users want simplicity; admins need full control and visibility

Solution Overview

Switchable Backends

Four retrieval modes — RAG Store, Vertex Search, Vector Search, or direct Gemini — switchable per-request, per-config, or per-environment.

Web-Based Config

PocketBase stores all RAG configurations (29 fields each). Changes take effect immediately — no restart required.

Two UIs

/ — Clean user chat. /admin — Full admin with backend selectors, config pickers, and source citations.

One-Command Deploy

Docker Compose orchestration with Caddy auto-HTTPS. Clone, configure .env, run setup.sh. Customizable branding per clone.

Technology Stack

FastAPI

Async Python API framework. Serves chat endpoint, admin CRUD, auth, and static UIs. Auto-generated Swagger docs at /api/docs.

PocketBase

Lightweight Go backend providing JWT auth, user management, RAG config storage, and chat history — all via REST API.

Caddy

Auto-HTTPS reverse proxy. Provisions TLS certificates via Let's Encrypt when a domain is configured. Zero-config SSL.

Docker Compose

Orchestrates all three services. One docker compose up starts the full stack. Local and production compose files included.

Google Cloud Vertex AI

Gemini LLM for generation. RAG Store, Vertex AI Search, and Vector Search for retrieval. Accessed via Python SDK.

Single-Page HTML UIs

Two self-contained HTML files — no build step, no JS framework. Inline CSS/JS served directly by FastAPI.

Retrieval Backends

None (Direct Gemini)

Chat with system prompt only. No retrieval. Ideal for general-purpose bots or when data is baked into the prompt.

backend: "none"

RAG Store

Vertex AI managed RAG corpus. Upload documents, automatic chunking and embedding. Similarity-based retrieval.

backend: "rag_store"

Vertex AI Search

Discovery Engine with query expansion, spell correction, boost specs, and structured snippets. Enterprise-grade search.

backend: "vertex_search"

Vector Search

Matching Engine with custom embeddings. ANN search, namespace filters, configurable leaf node fractions.

backend: "vector_search"

Switchable per-request, per-config, or per-environment — priority: request > PocketBase config > .env default

PocketBase Integration

User Management

  • Built-in users collection extended with role field
  • Two roles: user and superuser
  • JWT-based authentication
  • Registration and login via FastAPI endpoints

RAG Config Storage

  • 29 fields per configuration record
  • Covers all 4 backend types
  • One is_active config used as default
  • Changes take effect on next request — no restart

Chat History

  • Every query/answer pair stored
  • Tracks backend used, sources returned, config_id
  • Linked to user via relation

Auto-Migration

  • Collections created via JS migration on first boot
  • Schema versioned in pb/pb_migrations/
  • PocketBase admin UI at /pb/_/

Two User Interfaces

User UI — /

  • Clean, branded chat interface
  • Custom bot name, icon, welcome text
  • No backend selectors or admin controls
  • No source citations shown
  • Optional "Powered by" footer
  • Register / login / chat — that's it

Designed for end users

Admin UI — /admin

  • Backend dropdown (None / RAG / Search / Vector)
  • Config picker from PocketBase records
  • Collapsible source citations with scores
  • Backend badge in top bar
  • Access to stats API
  • Full RAG config CRUD via API

Designed for superusers

Both share the same auth system. Role determines access.

Configuration Management

Three levels of configuration, each overriding the previous:

1
.env defaults — Base configuration for the entire instance. Model, temperature, backend resource names.
2
PocketBase config — 29-field rag_configs record overrides .env per-setting. Active config used by default.
3
Per-request override — Pass backend, config_id, or top_k in the chat request body.

Key Parameters

LLM: llm_model, temperature, top_p, top_k, max_output_tokens, system_prompt

Backend-specific: corpus names, serving configs, filters, boost specs, embedding models, distance thresholds

Security

Authentication

  • PocketBase JWT tokens for all API calls
  • Tokens validated on every request
  • Registration creates user role by default
  • Superuser promotion via PocketBase admin only

Authorization

  • Role-based access: user vs superuser
  • Admin endpoints require superuser role
  • User UI hides admin controls entirely
  • Enforced at both UI and API layers

Secrets Management

  • .env file never committed to git
  • GCP credentials mounted as volume
  • deploy.sh excludes .env and credentials/
  • PocketBase admin password only in .env

Network

  • Caddy auto-HTTPS via Let's Encrypt
  • TLS certificates provisioned automatically
  • CORS middleware for local development
  • Internal services not exposed in production
